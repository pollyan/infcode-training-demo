# 故障注入说明

## 1. 这份文档的用途

这份文档帮助讲师更自然地演示：

1. 为什么要测试
2. 为什么要看终端日志
3. 为什么要用 `@终端`
4. 为什么最后还要做 Diff 审核

## 2. 推荐故障点

推荐使用：

**外部接口返回成功，但 `data` 为空或字段缺失。**

这类问题像真实联调问题，教学价值高，也容易触发空指针、映射错误或页面展示异常。

## 3. 推荐触发方式

### 场景一：字段缺失

客户编号：

`C202503002`

特点：

- mock 返回缺少 `contact_phone`
- 适合演示空值保护与兜底展示

### 场景二：上游异常

客户编号：

`C202503500`

特点：

- mock 返回上游服务异常
- 适合演示统一错误处理和 `@终端`

### 场景三：建议新增教学故障

客户编号：

`C202503998`

建议行为：

- HTTP 返回 200
- 但 `data` 为 `null`

教学价值：

- 很适合让学员理解“成功 HTTP 状态不等于业务结构完整”

## 4. 推荐演示顺序

1. 先跑 `C202503001` 成功路径
2. 再跑 `C202503002` 字段缺失
3. 最后跑 `C202503500` 或 `C202503998` 异常场景

## 5. 讲师提醒

1. 不要一出错就直接改代码
2. 先看日志，再定位在哪一层
3. 修完后一定要重新验证
4. 最后一定要回到 Diff 审核
