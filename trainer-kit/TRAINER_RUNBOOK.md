# TRAINER RUNBOOK

## 1. 这份文档是干什么的

这是一份讲师现场执行手册。

它不是项目说明，也不是学员任务单，而是帮助讲师在培训现场稳定推进节奏、及时纠偏、处理常见卡点的操作手册。

如果你准备带客户做一次完整演练，建议优先看这份文档。

---

## 2. 建议使用的分支

### `training/base`

用于培训现场演练。

特点：

- 项目骨架已具备
- 知识文档与规则已具备
- 但客户查询功能尚未完成

用途：

- 让客户在讲师带领下现场完成需求

### `training/done`

用于讲师备课、对照、兜底。

特点：

- 已包含一份可参考的完整实现

用途：

- 讲师提前熟悉结果
- 现场学员卡住时快速对照
- 培训结束后作为参考答案展示

## 4. 推荐开场话术

建议一开始就把今天的目标说清楚：

> 今天我们不是简单看一遍功能，而是一起走完一次真实研发任务。  
> 我们会先做上下文准备，再做需求理解和规划，然后再让 Agent 协助编码，最后还会做测试、排错和 Diff 审核。

这句话的作用是：

1. 先把客户从“看演示”切换到“做演练”
2. 先埋下“流程比单点功能更重要”的认知

---

## 5. 推荐现场节奏

## 阶段一：环境确认

目标：

- 让所有人进入同一个起点

讲师要做：

1. 打开项目
2. 说明今天用 `training/base`
3. 启动 mock、后端、前端
4. 打开首页

讲师要强调：

- 今天不是从零搭项目，而是在一个真实骨架上做需求

## 阶段二：建立上下文

目标：

- 让客户理解为什么不能一上来就写代码

建议引用：

- `docs/需求说明.md`
- `docs/外部集成说明.md`
- `.infcode/rules/`

然后通过 DeepMap Lite 自动生成：

- `Product.md`
- `structure.md`
- `tech.md`

讲师要强调：

- 企业场景里，先把资料和规则引进来，比写一个很花哨的提示词更重要

## 阶段三：先做需求理解

目标：

- 让 AI 先回答“会改什么”，而不是“直接改”

建议提问：

1. 会涉及哪些前端页面？
2. 会涉及哪些后端文件？
3. 会涉及哪些 DTO？
4. 外部集成要注意哪些异常？
5. 测试点有哪些？

讲师要观察：

- 学员有没有跳过规划，直接让 AI 改代码

如果有：

- 及时拉回
- 强调“先规划再执行”

## 阶段四：输出实施计划

目标：

- 让 AI 给出代码级任务清单

讲师要做：

1. 切到 Agent
2. 让它输出修改文件与步骤

讲师要检查：

- 是否识别了前端、后端、外部集成、异常处理
- 是否有明显越层设计

## 阶段五：先做后端

目标：

- 让客户先完成后端主链路

讲师要做：

1. 让 AI 先补 DTO
2. 再补 Controller / Service / Integration
3. 检查目录落位和分层

讲师要强调：

- “能跑”和“合规”是两回事

## 阶段六：再做前端

目标：

- 让客户补页面和查询调用

讲师要做：

1. 生成客户查询页
2. 生成前端 service
3. 跑通成功路径

讲师要观察：

- 页面里有没有直接写太多请求逻辑
- 错误提示是否友好

## 阶段七：测试与排错

目标：

- 演示真实开发里最有价值的一段：出问题以后怎么办

讲师要做：

1. 切换到故障场景数据
2. 运行后触发报错
3. 使用 `@终端`
4. 让 AI 分析根因和最小修复方案

讲师要强调：

- 不要一看到报错就直接改代码
- 先读日志，再判断发生在哪一层

## 阶段八：Diff 审核

目标：

- 把“AI 生成”重新拉回“人工可控交付”

讲师要做：

1. 打开代码变更列表
2. 看修改了哪些文件
3. 看有没有无关改动
4. 演示采纳与拒绝

讲师要强调：

- 企业研发最终要的是可审核、可回看、可交付

## 6. 现场最容易翻车的点

### 1. 学员直接让 AI 改代码

纠偏方式：

- 暂停
- 先让 AI 说明改动范围和计划

### 2. AI 直接把外部字段透传到前端

纠偏方式：

- 强调 DTO 映射
- 让学员重新检查规则文件

### 3. Controller 里写了过多逻辑

纠偏方式：

- 强调 Controller / Service / Integration 分层

### 4. 报错后只凭感觉修改

纠偏方式：

- 强制要求先看终端日志
- 用 `@终端` 做分析

### 5. 代码生成后直接结束

纠偏方式：

- 强制进入测试和 Diff 审核环节

## 7. 如果现场卡住，讲师怎么兜底

### 轻度卡住

症状：

- 学员思路乱
- AI 输出不够聚焦

做法：

- 帮学员把问题收窄
- 明确“这一步只做后端”或“这一步只做 DTO”

### 中度卡住

症状：

- 代码结构错了
- 文件落位错了

做法：

- 切到 `training/done` 对照讲
- 不必现场硬修所有细节

### 重度卡住

症状：

- 环境启动失败
- 多人无法继续

做法：

- 直接切讲师环境演示
- 保住流程体验，不纠结个别机器

---

## 8. 收尾时建议讲的结论

建议最后明确帮客户沉淀以下认知：

1. InfCode 的价值不只是生成代码，而是帮助研发流程变得连续且可控。
2. 企业场景里，正确的上下文输入比提示词花样更重要。
3. 规划、规则、测试、排错和 Diff 审核都不能省。
4. Agent 最适合处理多文件、有边界、有明确目标的任务。
