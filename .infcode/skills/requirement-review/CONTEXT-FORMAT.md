# CONTEXT.md 格式规范

本文件定义了项目 `CONTEXT.md` 的写作格式和规则。  
由 `requirement-review` Skill 在需求评审过程中，第一次确定项目术语时，按此格式懒惰创建客户项目的 `CONTEXT.md`。

> **重要：** 不要提前创建 CONTEXT.md。等第一个术语被确认后再建。

---

## 格式模板

```markdown
# <项目名称> 领域词汇表

<一到两句话：说明这个项目是什么，这个词汇表的适用范围。>

## 术语（Language）

**<术语1>**：
<一句话精确定义——说明它"是什么"，而非"做什么">
_Avoid（禁止替代）_：<同义词1>、<同义词2>

**<术语2>**：
<精确定义>
_Avoid_：<同义词>

## 术语关系（Relationships）

- 一个 **<术语A>** 包含多个 **<术语B>**
- 一个 **<术语C>** 属于唯一的 **<术语A>**

## 术语对话示例（Example Dialogue）

> **开发者：** "当 **<术语A>** 发生时，系统会自动创建 **<术语B>** 吗？"
> **领域专家：** "不会——**<术语B>** 只在 **<术语C>** 确认后才生成。"

## 已澄清的歧义（Flagged Ambiguities）

- "<模糊词>" 之前被同时用来表示 **<术语A>** 和 **<术语B>**——已解决：这是两个不同概念，分别用对应术语。
```

---

## 写作规则

1. **只收录项目特有的业务概念。** 通用技术概念（超时、错误类型、设计模式）不属于这里，即使项目大量使用。
2. **一个概念一个名字。** 如果同一事物有多种叫法，选最准确的那个作为规范术语，其余全部列入 Avoid。
3. **定义要精准，一句话为限。** 定义"它是什么"，不是"它做什么"。
4. **歧义要显式标出。** 发现一个词被模糊使用时，立即在"已澄清的歧义"里记录，并写明解决方式。
5. **展示关系，标注基数。** 用 **粗体术语** 表达术语之间的关系，明确一对多/多对一。
6. **用对话示例验证边界。** 对话是检验术语定义是否足够清晰的最快方法。
7. **CONTEXT.md 不是 Spec，也不是笔记本。** 不放实现细节、不放决策理由——那些属于 ADR 或 Spec 文件。

---

## 懒惰创建规则（Lazy Creation Rules）

- **不存在时**：当 Requirement Review 中第一个项目术语被确认，按上方格式创建 `CONTEXT.md` 并写入该术语。
- **存在时**：发现 Avoid 词汇立即标注 `❌ 矛盾: <词汇>`，新术语经 Developer 确认后立即追加，不批量处理。
- **其他 Skill 遇到 CONTEXT.md 不存在时**：静默跳过，不报错，不建议创建。
